# Changelog

## 0.1.0 — 2026-06-30 — AI rerank/gate pipeline

First versioned cut of the branch (`feat/identify-and-search-ai-pipeline`), prepared for merge to `main`.

### Added
- `BE/anthropic_captioning.py`, `BE/deepseek_captioning.py` — Anthropic (Claude) vision captioning/identification, and the DeepSeek text-only reranker (prompt + candidate formatting + result shaping self-contained in `deepseek_captioning.py`).
- `BE/openrouter_captioning.py` — `is_fish_image_openrouter()`, the production fish gate (Qwen3-VL-8B via OpenRouter). Reuses `_FISH_GATE_SYSTEM` + the JSON parser from `anthropic_captioning` so the gate prompt stays single-sourced.
- `BE/openrouter_captioning.py`: Qwen3-VL fish crop — `locate_fish_bbox_qwen()` (grounding: returns the main fish's bounding box in original pixel coords, downscaling to `CROP_SEND_MAX_EDGE` first so coords map back exactly) and `crop_b64_to_fish()` (pads extra on top for the dorsal fin, clamps, and returns the cropped JPEG base64; returns the image UNCHANGED on a null/tiny box so a localization miss never drops the fish). Adds `_CROP_SYSTEM` and the crop tuning constants (`CROP_MODEL`, `CROP_SEND_MAX_EDGE`, `CROP_PAD_FRAC`, `CROP_PAD_TOP_FRAC`, `CROP_MIN_AREA_FRAC`).
- `env-template`: `OPENROUTER_API_KEY`, `OPENROUTER_GATE_MODEL`, `OPENROUTER_CROP_MODEL` (both default `qwen/qwen3-vl-8b-instruct`).
- Provider management in `api_services.py`: `available_providers()` / `ACTIVE_PROVIDER` / `RERANK_PROVIDER`, conditional client construction (a missing key no longer crashes startup), `/changeModel` (cycles configured providers) and `/currentModel`.
- `fish_constants.py`: `PHYSICAL_FEATURE_FIELDS`, the 7-field caption prompts (`CAPTION_MATCH_*`), the shared details prompt `SYSTEM_CONTENT_DETAILS`, and the open-ended candidates prompt `SYSTEM_CONTENT_OPEN`.
- `env-template`: `IBM_COS_BUCKET`, `ANTHROPIC_API_KEY`, `ANTHROPIC_MODEL`, `DEEPSEEK_API_KEY`, `RERANK_PROVIDER`, `SPACE_ID`, `IBM_CLOUD_URL`.

### Changed
- `/identify_and_search` is now a full gate/rerank pipeline: new Qwen3-VL fish gate -> caption -> Elasticsearch kNN (top-10) -> new DeepSeek text rerank, returning the rich response (`reranked_results`, `observed_features`, `image_contains_fish`, `rejection_reason`). Image input remains a **COS object key**, now fetched via the shared `fetch_image_from_cos()` helper (extracted from the per-endpoint inline `cos.get_object` calls).
- The new fish gate (Qwen3-VL via OpenRouter) runs for **every** provider — a standalone call, independent of `ACTIVE_PROVIDER`, so switching providers never disables it. To use a different gate model, set `OPENROUTER_GATE_MODEL`.
- `/identify_and_search` now runs the fish gate and a Qwen3-VL crop localization **in parallel** (`ThreadPoolExecutor`, 2 workers) on the same image, so total pre-caption latency is `max(gate, crop)` instead of the sum. The gate stays authoritative for fish/not-fish (on a non-fish verdict the crop is discarded and the request short-circuits before the caption/search/rerank); on a fish verdict the crop tightens the image fed to captioning (`caption_input`) so the fish fills more of the vision model's fixed pixel budget. The crop is fail-safe: any crop error, or a null/too-small box, falls back to the full image, so it can only ever help. Both calls run for every provider; set `OPENROUTER_CROP_MODEL` to swap the crop/grounding model.
- The image sent to captioning is now capped to a `CAPTION_MAX_EDGE` (default 1344px) long edge via `cap_image_b64()`, applied on both captioning paths (`/identify_and_search` Pass-1, after the crop; and `/image_captioning`). The vision model downscales larger images anyway (Sonnet 4.6 clamps to ~1568px / ~1.15MP before billing), so capping at 1344 trims upload bytes and billed vision tokens (~25% fewer than the 1568 ceiling) with negligible detail loss. Downscale-only and fail-open: it never upscales, and on any parse/encode error or an already-small image it returns the original bytes unchanged. Set `CAPTION_MAX_EDGE=0` to disable. The identification endpoints (`/image_identification`, `/search_possible_fish`) are NOT capped.
- `/search_possible_fish`, `/image_identification`, `/image_captioning` now branch on `ACTIVE_PROVIDER` (Anthropic/Gemini/WatsonX/Groq) while keeping the COS-key entry point.
- Normalized the details identification prompt: the four per-provider prompts (anthropic `_DETAILS_SYSTEM`, watsonx `system_content`, groq `SYSTEM_CONTENT_FULL`, gemini `system_content_full`) are replaced by a single shared `SYSTEM_CONTENT_DETAILS` in `fish_constants.py` (structure follows the Anthropic version). This fixes an inconsistency where `/image_identification` returned `physical_description`/`habitat` in a language that depended on the active provider (Thai for groq/gemini, English for anthropic/watsonx). Now consistent across all providers: `physical_description` + `habitat` are written in **Thai** (names/scientific/order stay English) for the Thai-facing app, including the gemini `response_schema` field descriptions. Response shape unchanged.
- Normalized the identification provider dispatch: `/image_identification` and `/search_possible_fish` now resolve the provider through registries (`IDENTIFY_DETAILS_FUNCS` / `IDENTIFY_CANDIDATES_FUNCS`) of uniform `(b64) -> dict` adapters instead of hand-rolled `if/elif/else ACTIVE_PROVIDER` chains. Same providers, same response shapes (no frontend change); adding/swapping a provider is now one adapter + one map entry. An unknown `ACTIVE_PROVIDER` now fails explicitly (500 `unknown provider`) instead of silently falling through to Groq. The watsonx candidates adapter does its `get_watsonx_token()` pre-step internally. Captioning dispatch (`/image_captioning`, `/identify_and_search` Pass-1) is unchanged.
- `/generation` (the `/chat` page) now runs on **DeepSeek** (`deepseek-v4-flash` via `chat_deepseek`, model from `DEEPSEEK_MODEL`) instead of WatsonX `meta-llama/llama-3-3-70b-instruct`. `generation.py` no longer imports `ibm_watsonx_ai` for chat (the WatsonX `ModelInference` init was removed); needs `DEEPSEEK_API_KEY`. Note: chat and the rerank share the `DEEPSEEK_MODEL` env (both default `deepseek-v4-flash`).
- `/search_possible_fish` (and the `/identify_and_search` open-set fallback, which shares the same candidate functions) now make an **open-ended** species guess instead of being constrained to the 91-species CSV. All four provider candidate functions switched from `SYSTEM_CONTENT_SINGLE` (allowed-list + reference-description comparison) to a new `SYSTEM_CONTENT_OPEN` prompt that lets the model name the actual species it believes the fish is. The response shape is simplified — `observed_features` was dropped from the candidates output (the gemini `response_schema` too); each result is `{fish_name, score, score_reason}` as before. This is the correct behavior for the out-of-corpus path: a fish flagged as *not* in the database is no longer forced to match one of the 91.
- `requirements.txt`: added `anthropic`; removed a duplicate `ibm-cos-sdk` line.

### Notes
- The fish gate runs for any provider (needs `OPENROUTER_API_KEY`); a gate failure is non-fatal and falls through to captioning. The rerank runs whenever a caption provider is configured (needs `DEEPSEEK_API_KEY`); if DeepSeek is unavailable or errors, it degrades to Elasticsearch-only results.
- COS bucket defaults to `fish-image-bucket` (`IBM_COS_BUCKET`); keep this aligned with the frontend's upload bucket.
- **TODO (next implementer): wire up the open-set `possible_new_species` flow on the frontend.** `/identify_and_search` already returns the open-set verdict fields — `match_confidence`, `confident_match`, `possible_new_species`, `suggested_endpoint` (`/search_possible_fish` when low-confidence), and `open_search_results` (populated only when `IDENTIFY_AUTO_FALLBACK=1`). The `thai-seatizen` frontend does **not** consume these yet (its `/identify_and_search` zod schema in `packages/ai/lib/fetcher.ts` omits them, so they're silently dropped), meaning a flagged out-of-corpus fish still surfaces as a normal match.
